## Context

The synced map+list on `/` (just shipped, commits `b3d6b74..e9c9639`) keeps the
homepage state-rich: scroll position, map view, sort context. Clicking a list
card or marker today navigates to `/park/<slug>`, blowing all of that away. The
user has been using competitor map-tools recently and wants the Zillow/Airbnb
pattern: park details open as a modal layered over the homepage. URL updates to
`/park/<slug>` for shareability, but the user never leaves `/`. Back button or
explicit close returns the homepage to its prior state intact.

`/park/<slug>` remains a real standalone page. Direct hits (refresh, search
results, shared links, Googlebot) get the full page. SEO is structurally
preserved by Next.js's intercepting+parallel routes pattern — the same URL
serves a modal from `/` and a full page on direct access.

## Vision

### Approach A — Intercepting + parallel routes

The Next.js canonical "Instagram photo modal" pattern. Folder structure (this repo uses `src/app`, not root `app`):

```
src/app/
  layout.tsx              ← takes {children, modal}
  page.tsx                ← homepage (existing)
  park/[slug]/page.tsx    ← unchanged — full standalone page
  @modal/
    default.tsx           ← returns null when nothing to show
    (.)park/[slug]/
      page.tsx            ← server component: data fetch + metadata
    _components/
      ModalShell.tsx      ← 'use client' wrapper for <dialog> lifecycle
```

- **Layout change:** root `app/layout.tsx` becomes `<SiteHeader />{children}{modal}<Footer />`. The `{modal}` slot is null on every page except when an intercepting route matches.
- **Server/client boundary:** the intercept route's `page.tsx` is a server component (data fetch + metadata). It imports and renders a client component `ModalShell` (`'use client'`) which owns the `<dialog>` ref, `showModal()` call, ESC/backdrop close, and `router.back()`. Server fetches the park, passes it as a prop to ModalShell, ModalShell renders `<dialog>` + `<ParkProfile>`. Clean split; no muddled boundary.
- **`generateMetadata` is shared via helper (eng review D3):** the standalone route at `src/app/park/[slug]/page.tsx` and the intercept route at `src/app/@modal/(.)park/[slug]/page.tsx` both call a new `parkMetadata(slug)` helper (in `src/lib/park-query.ts` next to `getParkBySlug`). Single source of truth for title format, description fallback, and any future OG image / JSON-LD additions. Without this, Next.js applies metadata from the matched route, so the intercept route MUST export `generateMetadata` for the tab title to update — the helper avoids the duplication that would otherwise drift.
- **`notFound()` inside the modal route** — if `getParkBySlug` returns null (stale data, mistyped URL), the modal route renders a `<dialog>` containing a "Park not found" message + close button. NOT `notFound()` from Next, because that would 404 the entire homepage shell. Standalone path's `notFound()` is unchanged.
- **`<dialog>` element** with `.showModal()` — gets focus trap, ESC handling, document scroll lock, ::backdrop for free (browser-native). Note: `<dialog>` scroll-locks the document but NOT arbitrary nested scroll containers, so `SyncedMapList`'s list pane (which has its own `lg:overflow-y-auto`) retains its scroll position — which is exactly the point.
- **Component reuse:** `ParkProfile` (76 LOC, 16 sections) is already chrome-free — the `<main>` wrapper and `<Breadcrumb>` live in `app/park/[slug]/page.tsx`, not inside ParkProfile. Standalone page keeps both. Modal route uses `<dialog>` + close button + ParkProfile. No component splitting needed.
- **Breadcrumb / JSON-LD:** stays on standalone page only. Modal version doesn't render Breadcrumb → no JSON-LD inside modal → no duplicate JSON-LD risk across two contexts. Google only sees the standalone page.
- **Mobile (<1024px):** modal sized to full viewport (`width: 100vw; height: 100dvh; margin: 0; border: 0; border-radius: 0`). Top-left back arrow (chevron + "Back" or just icon) closes the modal. Looks like a new page.
- **Desktop (≥1024px):** modal centered, max-width ~640px (matches existing `max-w-2xl` chrome), max-height ~90dvh, scrollable interior. Click ::backdrop or X (top-right) to close. Click outside via small `onClick` handler on `<dialog>` that checks `event.target === dialogRef.current`. Close action calls `router.back()`. **Close-target fallback:** if `router.back()` would leave the site (e.g., user landed on `/park/<slug>` directly, then navigated client-side somewhere odd — unlikely but possible), prefer `router.push('/')` as a safer fallback. Check `window.history.length` or use a guard.
- **Strategic content shape (D6.4 — locked):** ship the FULL `ParkProfile` (all 16 sections) inside the modal. Mobile = page-equivalent (matches user intent: "looks like a new page"). Desktop = scrollable 640px modal. If the desktop scroll feels cramped after shipping, the follow-up is a compact summary view (TODOS). Don't pre-build the compact version.
- **History stack:** Next.js's intercepting route auto-pushes a history entry on navigation. Browser back button → modal closes naturally. No manual history management.
- **Modal-to-modal navigation: NOT SUPPORTED (D6.1 — locked).** `<dialog>.showModal()` makes everything outside the dialog `inert` — the list pane and map are unclickable while the modal is open. User must close the modal first to click another park. This is acceptable UX (matches Airbnb, Zillow, most modern sites). No `router.replace` conditional, no NearbyCard client-component complexity. Simpler.
- **Map popup intercept fix (D6.2 — locked).** `MapPopupContent.ts` currently uses `document.createElement('a')` with an `href` — browser-native navigation, which BYPASSES the intercept route. Fix: convert the popup-link click to a router-aware navigation. Smallest viable: add `link.addEventListener('click', (e) => { e.preventDefault(); router.push(href); })` where `router` is passed in (or use Leaflet's popup factory closure to access the parent's router). The popup itself still renders the same HTML; only the click handler changes. ~10 LOC delta in MapView/MapPopupContent.
- **List card intercept fix (D6.2 — locked).** `NearbyCard.tsx` currently renders `<a href={item.href}>`. Swap `<a>` → `<Link>` universally. Used in 6 places (homepage, park profiles' Nearby Parks/Shops, county pages, obstacle pages); swap is a universal speed/prefetch improvement and required for any /park/<slug> navigation from the list to hit the intercept. No conditional behavior needed (modal-to-modal swap is gone per D6.1).
- **`.card-selected` lifecycle (single derived state):** the existing `popupOpenForId` effect in `SyncedMapList` (which owns `flashedElRef`) is refactored — NOT supplemented — to drive off a derived `selectedId`. Pseudocode:
    ```ts
    const modalParkId = useMemo(() => {
      const m = pathname.match(/^\/park\/([^/]+)$/);
      if (!m) return null;
      return parks.find((p) => p.slug === m[1])?.id ?? null;
    }, [pathname, parks]);
    const selectedId = modalParkId ?? popupOpenForId;
    // existing effect now depends on selectedId, not popupOpenForId
    ```
  This avoids two effects fighting for `flashedElRef`. Precedence is `modalParkId` over `popupOpenForId` because the modal is a stronger commitment than a hover popup. Note: while modal is open, hover doesn't fire (list is inert per D6.1), so in practice popupOpenForId is null when modalParkId is set — but the precedence rule still matters for the transition frames.
- **Suggest-an-Edit nested modal (D6.3 — corrected from codex; mechanism tightened by eng review D2):** `SuggestEditModal` is a `<div role="dialog">`, NOT a native `<dialog>`. The plan previously claimed top-layer stacking; that's wrong. Actual behavior: outer native `<dialog>` makes its siblings inert but NOT its descendants. The `SuggestEditModal` div, rendered inside `ParkProfile` (which is inside the outer dialog), stays interactive. Stacking works via z-index, not top-layer. ESC handling: when ESC fires, two things happen in parallel — (1) `SuggestEditModal`'s document keydown listener (line 41-46) calls `onClose()`, and (2) the browser's user-agent default action fires a `cancel` event on the outer `<dialog>`. The plan previously said "stopPropagation" — that only stops JS event propagation, NOT the UA cancel. The correct fix: `SuggestEditModal`'s keydown handler must call `e.preventDefault()` (suppresses the dialog's UA default action) AND `e.stopPropagation()` (belt-and-suspenders), AND guard so it only fires when SuggestEditModal is itself the topmost open layer. Lightbox (PhotoGallery) is ALSO a native `<dialog>` inside ParkProfile — when both park modal and Lightbox are open, the browser's native top-dialog ESC handling closes Lightbox first; no JS intervention required for the all-native nesting.
- **`<Link prefetch>` (D3.3 — accepted):** audit that list cards use `<Link href="/park/<slug>">` not `<a href>`. Next.js prefetches the matched route on hover by default — modal data is warm by the time the user clicks. ~0 LOC if already using `<Link>`; quick swap if not.
- **Focus return on close:** the `<dialog>` element doesn't restore focus to the trigger automatically. For v1 we accept that focus returns to `<body>` after close (browser default); user can Tab from there. The "ideal" capture-trigger-and-restore requires either intercepting the card click handler (server-component compatibility issues) or sessionStorage (gross). Deferred to TODOS.md.
- **Animation:** `<dialog>` toggles `display: none` via `[open]`, which short-circuits CSS transitions. A real fade/slide needs `@starting-style` + `transition-behavior: allow-discrete` (Safari < 17.5 silently ignores). For v1 we ship NO animation — modal appears/disappears instantly. Honors `prefers-reduced-motion` trivially. Deferred polish to TODOS.md.
- **`default.tsx` is load-bearing:** `src/app/@modal/default.tsx` returning null is REQUIRED. Without it, Next.js 404s the @modal slot on every non-intercepted route, which means every non-/park page would render a "default missing" error. Not optional.
- **A11y — modal label (D4):** `<dialog>` gets `aria-labelledby="park-name"`, reusing the existing `<span id="park-name">{park.name}</span>` that ParkProfile renders for its standalone-page `aria-labelledby`. Screen readers announce the park name on open. Zero new markup.
