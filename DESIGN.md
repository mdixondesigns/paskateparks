### What `/plan-eng-review` should focus on

This design doc locks **what** to build. Architecture is what's open. Eng review should produce decisions on:

- **Data import pipeline** — WordPress (3 custom post types `park` / `builder` / `shop`, 3 custom taxonomies) → Airtable. One-time bulk import + ongoing content authoring for the 99 stub parks. Lat/lng extraction specifically requires WP DB export, not HTML scraping (see SITE-AUDIT.md §9 #2).
- **Image hosting** — current photos live at `paskateparks.com/wp-content/uploads/`. Migrate to where? Airtable attachments (storage limits)? Cloudinary? S3 + CDN? Phased migration with WP as transitional source?
- **Airtable → Next.js build pipeline** — ISR with `revalidatePath` triggered by a Vercel deploy hook (per original §Distribution Plan). Confirm Airtable rate-limit handling for the single paginated list call at build time.
- **Build-time compute** — Haversine for Nearby Parks (3 closest within 30mi) and Nearby Shops (same, no state border rule per D7). Per-park computed values stored in page props.
- **Suggestions backend** — D28 modal on park profile → Airtable Suggestions table. Anti-spam strategy (rate limit, honeypot, captcha)? Submission auth (anonymous OK)? Moderation workflow.
- **410 Gone routing** — 14 `/builder/<slug>/` + 20 `/shop/<slug>/` URLs need HTTP 410 in `next.config.js` (per D3). Confirm Next.js routing approach.
- **Status banner UX** — D11: Permanently Closed parks keep their pages live but should communicate closure prominently. Banner spec, dismissibility, visual weight.
- **Photos child table query** — D29: Photos is a child table linked to Parks. Confirm Airtable linked-record fetch + Order-based sorting at build time.
- **Programming module** — D27: conditional render mid-page when `Programming = true`. CTAs are template-level, not per-park. Confirm template handles the toggle cleanly without layout shift.
- **CI/CD** — Vercel GitHub integration auto-deploy on merge. Airtable webhook → Vercel deploy hook on record save. Confirm.
- **Performance budget** — mockup is mobile-first; original premise is "parent in a parking lot on an Android." Target: TTI under 3s on 4G, LCP under 2.5s.

### 2026-05-30 — Session 2 (post site audit)

After the live site audit ([SITE-AUDIT.md](docs/archive/SITE-AUDIT.md)), the following decisions extend or refine this design. Original sections below remain as historical context; this section is the current source of truth where it conflicts.

#### Park scope clarification

**All 150+ parks will have their own profile pages.** The 47-of-146 ratio on the live site reflects unfinished content authoring, not a tiered strategy. The rebuild template must render gracefully when fields are missing (description, photos, obstacles, etc.) — data will be filled in over time. Same graceful-degradation rule as the builder/shop sections: hide empty sections silently, render available data, never show placeholder copy like "coming soon."

#### Builders + Shops — 8 decisions locked

1. **No standalone pages.** Builders and shops do not get their own URLs. They appear only as contextual sections on park profiles.
2. **Single curated outbound link per business.** Owner picks the best link — official website or active social (Instagram/Facebook). Inactive social accounts excluded by owner judgment.
3. **Existing 34 URLs → 410 Gone.** All `/builder/<slug>/` and `/shop/<slug>/` URLs return HTTP 410 in `next.config.js`. Tells Google these are permanently deleted. No redirect maintenance.
4. **Builders section on park profile — multi-builder supported.** Renders as comma-separated outbound credits ("Built by Spohn Ranch Skateparks → 5th Pocket Skateparks →"). Joint builds get full attribution.
5. **Nearby Shops on park profile — top 3 within 30 miles.** Build-time Haversine computation against shop lat/lng. Section hides silently if 0 in range. **No state border rule** — Cleveland, Trenton, etc. eligible if they're the closest.
6. **Empty link state = plain text.** Records with name but no curated link show as text only. Same rule for builders and shops. Handles DIY ("Built by DIY"), new shops awaiting curation.
7. **Optional logos for both builders and shops.** Logo + name side by side inside one outbound link. No logo → name only. Same accessible fallback regardless.
8. **Forward-looking:** Shops on map — `Lat`/`Lng` required from day 1 so the feature can flip on without schema migration. Builders on map — not planned (orgs, not locations).

#### D29 — Photos child table (replaces the single Photos attachment field on Parks)

| Field | Type | Required |
|---|---|---|
| `ParkRef` | Linked records → Parks | ✅ |
| `File` | Attachment (single) | ✅ |
| `Credit` | Single line text | — |
| `Caption` | Long text | — |
| `Order` | Number | — |

Implications:
- The Parks `Photos` field becomes a relation; renderer queries Photos by `ParkRef`, sorts by `Order`.
- **Amenity photos** (`BathroomPhoto`, etc.) stay as simple attachments on Parks — owner-sourced, no contributor credit needed.
- **Builder and Shop logos** stay as simple attachments — single logo per record.
- **Accessibility:** alt text auto-generates from park name + photo index when `Credit`/`Caption` are absent (e.g., `"FDR Skatepark photo 3"`). Owner-supplied `Caption` overrides; `Credit` appears as a small attribution below the photo in the lightbox.

#### Deferred to future iterations

- **Park rating system (1–10 score)** — discussed and explicitly parked 2026-05-30. v1 ships as an impartial directory (Wikipedia-style). Future iterations may introduce an owner score + community reviews + computed-signal blend. Architecture options analyzed in session conversation: (A) single owner score + computed signals as separate filters + future community reviews table; (B) computed composite from existing signals; (C) multi-dimensional scores (parent/skater/overall). No fields added to v1 schema for rating. If revisited: candidate fields include `OwnerScore`, `OwnerScoreNotes`, `OwnerScoreDate`, `Condition`, `SkillLevel`, `Notable`, plus a `Reviews` child table.
